#1 middleware.ts  — MISSING 0 / 10 → 9 / 10
+12pt
▶

Issues Found

CRITICAL File does not exist — all routes are publicly accessible
Without middleware.ts, Clerk's clerkMiddleware() never runs. Every /api/* route and every /(dashboard)/* page is fully unauthenticated. Any user can hit the agent route, read thread state, and trigger write actions — no credentials required. This is the single most severe gap in the codebase.
CRITICAL RBAC context headers never get injected
rls-client.ts reads x-current-org-id, x-current-user-role, etc. from request headers — but those headers are only set by middleware. Without it, getContextFromHeaders() always returns null, so the RLS wrapper silently falls back to the wrong identity.
HIGH Dashboard pages not protected — no redirect to sign-in
Unauthenticated visitors can browse /chat, /insights, /admin directly. Clerk's server-side auth() calls inside those pages will throw rather than redirect.

Prioritised Fix Plan

  • P1 Create middleware.ts using clerkMiddleware(), protect /api/* and /(dashboard)/*, public routes: /, /sign-in, /sign-up
  • P2 Inside middleware, after auth, call resolveUserAccess() and inject RBAC headers onto the response object so downstream handlers can use getContextFromHeaders()
  • P3 Add a matcher config so middleware doesn't run on /_next/static or /public assets (performance)

// middleware.ts
// *** FILE DOES NOT EXIST ***

✓ AFTER — production-ready
import { clerkMiddleware, createRouteMatcher }
  from '@clerk/nextjs/server';
import { NextResponse } from 'next/server';
import { resolveUserAccess } from '@/lib/auth/rbac';

const isPublic = createRouteMatcher([
  '/', '/sign-in(.*)', '/sign-up(.*)'
]);

export default clerkMiddleware(async (auth, req) => {
  if (isPublic(req)) return;
  const { userId, orgId, orgRole } = await auth.protect();
  const access = await resolveUserAccess(
    userId, orgId, orgRole
  );
  const res = NextResponse.next();
  res.headers.set('x-current-org-id', orgId);
  res.headers.set('x-current-user-id',
    access.internal_user_id ?? '');
  res.headers.set('x-current-user-role',
    access.role ?? 'member');
  res.headers.set('x-current-user-dept-id',
    access.dept_id ?? '');
  return res;
});

export const config = {
  matcher: ['/((?!_next/static|_next/image|favicon).*)' ]
};

60 seconds means 1 Supabase query per minute per active user. At 100 concurrent users this is 100 DB calls/min just for RBAC. Consider 5 minutes with an explicit invalidation on role change (per the spec which says 5 min TTL).

Issues Found

CRITICAL No input validation — message is passed raw to the LLM
const { message, threadId } = await req.json() with zero validation. If message is undefined, new HumanMessage(undefined) throws an opaque error. If it's 10 MB of text, it burns tokens and may crash the process. At minimum: require non-empty string, enforce max length.

HIGH Thread ownership not verified before resuming graph state
Any authenticated user can pass any threadId and read another user's graph state. The status route does verify ownership, but the main agent route does not — it just calls graph.stream(initialState) with the user-supplied ID.
MEDIUM Rate limiting absent
A single user can spam the agent endpoint. Each request spins up a LangGraph stream and makes Anthropic API calls. Without rate limiting, one bad actor can run up the platform's LLM bill or DoS the service.

  • P2 Return "Internal Server Error" literal for 500s, log the real error server-side only
  • P3 If threadId is provided, verify it belongs to userId/orgId before streaming
  • P4 Add Redis-based rate limit: e.g. 20 requests/user/minute using incrWithExpire

#5 lib/qstash/verify.ts 6 / 10 → 9 / 10
+4pt

Issues Found

HIGH url param omitted from receiver.verify() — replay attacks possible
QStash signatures include the destination URL. Without passing url: req.url to receiver.verify(), an attacker can capture a valid signed payload from any worker endpoint and replay it against a different worker endpoint (e.g. replay a "briefing" job as a "data-index" job). The url field is mandatory for full protection.
HIGH Empty signing keys fall through silently
process.env.QSTASH_CURRENT_SIGNING_KEY || '' — an empty string is accepted as a key value. If the env var is missing in production (e.g. misconfigured deployment), the receiver is initialised with empty keys and may accept or reject everything unpredictably.

Prioritised Fix Plan

  • P1 Pass url: req.url to receiver.verify()
  • P2 Guard the env vars — throw at module load if either signing key is absent
✗ BEFORE — URL missing
const isValid = await receiver.verify({
  signature,
  body,
  // ← url NOT passed
  // replay attacks possible
});
return isValid;
✓ AFTER — full verification
const isValid = await receiver.verify({ signature, body, url: req.url, // ← required! }); return isValid;
